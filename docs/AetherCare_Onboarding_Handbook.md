## 0. 為什麼存在這份文件

新員工最常見的失敗不是不知道原則，是**在壓力下覺得「這次例外」**。

業績壓力、投資人壓力、合作案壓力、競品壓力、客戶要求、團隊情緒——任何一個都可能讓我們偷偷把產品改回「比較好賣但承諾比較假」的版本。

這份 handbook 不是 mission deck。它是一份**硬約束**：當壓力來臨，你打開這份文件，找答案，然後決定。它的目的是讓 thesis 開始阻止你做錯事。

---

## 1. 我們做什麼，一句話

> **AetherCare 替已經願意承擔的人，留下他們需要的事實，讓家庭照護不要壞得更快。**

不是：
- 守護長者
- 用 AI 陪伴長者
- 救命系統
- 讓家庭更靠近
- 修復家庭關係
- 智慧照護平台
- 長照前台

是：
- 家庭照護事件的**最小可追蹤協議**
- 讓「狀態、承接、轉交、結案、反制、回填」不被吞掉
- 讓家庭在需要外部協調時，**有一份可攜帶的照護事實**

---

## 2. 我們的承諾邊界

| 承諾 | 不承諾 |
|---|---|
| 照護事實不被吞掉 | 救命 |
| 責任不被洗掉 | 家庭變親近 |
| 外部資源看得懂發生過什麼 | 修復多年家庭衝突 |
| 「不用從頭解釋」 | 把孤獨變成被愛 |
| 「比較不壞」 | 「真的好起來」 |

> **產品能改善家庭運作，不等於能改善家庭情感。**
> 這是團隊每一個 PM、設計師、工程師、業務都要內化的一句。

---

## 3. 5 條設計原則（資料層）

1. **每個 alert 必須產生不可省略的事件鏈**：送達 → 看見 → 承諾行動 → 結案 → 事後結果。UI 可柔化措辭，**資料層不得合併到失去責任判讀能力**。
2. **ACK 只能表示「我看見」，不能表示「我負責」**。結案前必須有明確行動狀態：「正在聯絡」「已到場」「無法處理，轉交」。
3. **結案必須綁定個人身份、理由碼與時間戳**。不允許家庭群組、匿名角色或「已處理」這種無主語狀態完成閉環。
4. **長者反制權預設存在，但不要求每案簽核**。反制是結案後的異議流程，不是即時投票。
5. **匯出資料必須分用途**：家庭日常視圖最小化指責語氣；外部協調視圖保留完整事件軌跡；每次匯出留下用途、範圍、操作者紀錄。

---

## 4. 12 條反功能清單

**短期內絕不做**。每次 roadmap review 都要對照這份清單。任何一條若被排進 sprint，必須由創辦人 + 產品負責人簽字。

1. 陪伴 AI / 對話機器人 / 虛擬伴侶
2. 跌倒偵測硬體 / 床墊 / 雷達整合（BYOD 平台）
3. 家庭協商頻率 KPI / 互動頻率排行榜
4. 家屬回應速度排名 / 孝順分數
5. AI 改寫長者原話 / 情緒分析 / 溫柔摘要
6. 月度家庭關注摘要（事件摘要 OK，「關注度」不 OK）
7. 保險公司風險分 / 核保資料合作
8. 政府入口式 onboarding（成為長照前台）
9. 節慶 / 孝順節日推播
10. 全量 BYOD 開放平台（早期只接最多一個 reference integration）
11. 「我們替長者監測」式的 marketing 文案
12. 在 UI 層使用「失職」「未回應」「責任」等審計詞彙

## 5. 5 層產品架構（Round 8 收斂版）

```
L0 感知層 — 砍到最薄
   只保留：長者 check-in、家屬事件回填、LINE/SMS/app 通知狀態
   最多一個 reference integration，不做 BYOD

L1 狀態層 — 結構化事實，不做 AI 敘事
   每日狀態：未結事件數、最近回應時間、coverage gap、裝置沉默
   絕不寫「媽今天看起來 OK」這種 AI 安撫句

L2 責任協議層 — 唯一真正的 MVP 核心
   送達 → 看見 → 承諾 → 轉交 → 結案 → 反制 → 回填
   on-call 排程：讓 caregiver 可預先聲明「目前無法承接」

L3 證據可攜帶層 — 延後做
   先做原始事件鏈匯出 + 家庭內部事實摘要
   pilot 後再決定外部協調視圖、長照評估表相容、爭議匯出格式

L4 外部資源提示層 — 不做 routing
   只列資源清單（1966、長照中心、居服申請、社工資源）
   不主動判定「你們不夠了」、不做媒合、不做資格判定

橫切：
- 承諾邊界標示：每個狀態與輸出標示「事實 / 推論 / 回填 / 未確認」
- 反規避設計：不羞辱、不審判化、不讓 ACK 看起來像法律證據
```

## 6. 對外語氣 vs 對內資料：分層原則

**核心原則**：語氣要柔，契約要硬。**反過來做會更慘**：UI 很柔、資料很狠，使用者會覺得被騙。

| 對內資料層（保留責任語義） | 對外 UI 用詞（協作語義） |
|---|---|
| `responsibility_chain` | 接力紀錄 |
| `failed_to_acknowledge` | 目前無法承接 |
| `closed_by` | 目前由誰跟進 |
| `unresolved_escalation` | 等待下一位 |
| `accountability_export` | 家庭照護摘要 |
| `caregiver_off_call` | 暫時不在 on-call |

**這不是洗掉責任，是分層治理**。差別在：**可不可在必要時還原事件鏈**。若不能還原 = 洗掉責任；若能還原但日常不拿來羞辱 = 分層治理。

## 7. 我們服務誰，不服務誰

**服務（target）**：
- 家庭仍存在、但照護協調已開始失靈
- 已經發生過「我以為你會處理」這類失誤的家庭
- 想承接但抗拒被審判的家屬（**台灣市場主流**）
- 長者還能講話、有手機、能表達異議

**不服務（明確拒絕）**：
- 完全獨居、無任何家屬或熟人能負責的長者 → 引導到 24h 客服 / 居服員 / 社區照顧
- 完全健康、家屬只是「想多看一眼」 → 引導到 LINE 即可
- 已入長照機構的長者 → 機構有自己的班表系統
- 全家拒絕承接、想用產品監控長者的子女 → 拒絕
- 拒絕長者可見性、要求刪除責任紀錄的客戶 → 拒絕

**集體拒絕加入的家庭不是 onboarding 失敗，是 acquisition signal**：這個家庭沒有可被協議化的照護責任。不要花銷售成本追。

## 8. 決策 North Star — 未來 5 年只問這一個問題

當你不確定某個功能、合作、客戶、補助、投資條件該不該做時：

> **這會讓照護事實更可交接，還是讓我們承諾一個自己無法承接的結果？**

- 答案是「更可交接」→ 繼續問隱私、權限、語氣與資料治理
- 答案是「承諾無法承接的結果」→ **砍掉，無論它多好賣**

## 9. 5 年內最危險的 3 個誘惑（提前知道，提前免疫）

### 誘惑 1：保險公司合作
**他們會說**：帶來客戶、補貼費用、提高信任。
**真正的代價**：責任鏈從「家庭交接工具」變成「風險評分材料」。長者感覺被核保，家屬開始管理紀錄外觀。**照護事實變成可被保險解讀的行為資料**。

### 誘惑 2：政府計畫補助
**他們會說**：提供正當性、資源、合法性。
**真正的代價**：產品慢慢從「家庭責任協議」變成「長照流程前台」。**服務更多人，但每一個人的問題都處理得更淺**。

### 誘惑 3：AI 大模型新功能
**他們會說**：新模型能做情感摘要、語音陪伴、家庭會議摘要、健康推論。
**真正的代價**：**Demo 很漂亮，承諾很滑**。AetherCare 需要的 AI 是不確定性標記、矛盾偵測、交接整理；**不是冒充家人**。

**被收購提案**屬於同一類：若收購方要的是責任鏈資料去餵保險、硬體監測、廣告或政府風控——**那不是 exit，是 thesis 被解體**。

## 10. 三個最容易讓創辦人 / 團隊動搖的情境（預警）

| 情境 | 對方會說 | 你內心會想 | 正確反應 |
|---|---|---|---|
| 家屬說「太冷」 | 你們這樣好像在追責，不像照護 | 加溫暖文案、情感摘要 | 可以調語氣，**不能改資料結構** |
| 投資人說 TAM 太小 | 你們為什麼不做陪伴 / 硬體 / 保險 / 政府 | 也許可以加一點點 | TAM 變大時，**承諾通常也變假** |
| 合作方拿錢要求越界 | 我們可以付錢，只要你們加一個 X 功能 | 錢來了不收很可惜 | **錢最常讓邊界看起來像保守，而不是保命** |

## 11. 「壓力下不改口」測試

每季一次，找一個團隊外的人，要求他檢查以下 4 個介面：

1. **官網首頁** — 有沒有偷出現「守護」「陪伴」「安心」「讓愛不缺席」
2. **App store 描述** — 有沒有偷說「救命」「緊急救援」
3. **投資人 deck** — 有沒有偷把承諾講得比能做到的大
4. **業務銷售腳本** — 業務在外面實際怎麼介紹產品

**若任何一處偷回去，代表邊界沒有真的被接受**。需要團隊重讀這份 handbook。

## 12. 我們的內部 KPI（只看 4 個）

- **事件鏈完整度**：每個事件是否有承接者 + 結案者 + 事後回填
- **長者反制使用率**：不是越低越好，使用率 = agency 真實
- **家庭留存**：design partner 60-90 天後還在用嗎
- **使用者原話**：是否自發說出「不用從頭解釋」「我以為你會處理」這類語言

**不看**：DAU、活躍家屬數、ACK 率、回應速度排名、家庭協商頻率、推播打開率。

---

## 13. Pilot 失敗判準（先寫死，避免日後找藉口）

20-30 個精準家庭 pilot，60 天觀察。如果連 5-7 個都無法形成可用責任鏈，**先不要怪市場**，先檢查：

- 不知道產品在記錄什麼 → onboarding 錯
- 覺得像被審判 → 語氣與權限錯
- 事件鏈太麻煩 → 流程錯
- 沒有足夠真實事件進來 → 客群選錯
- 有事件、有混亂、有需求，但仍沒人願意留紀錄 → **才是 thesis 錯**

---

## 14. 我們仍然不知道的事（誠實標記）

這 11 輪辯論**沒解決**的問題，需要 pilot 真實驗證：

1. 台灣家庭中「願意照顧但不願被審計」的比例到底多高？這決定 TAM 是否足夠
2. 長者代理性提高後，付費子女購買意願會下降多少？下降的是錯客戶還是核心市場
3. 「外部資源轉介提醒」在真實家庭裡會被理解成幫助、羞辱、推責、還是廣告
4. AI 不確定性標記會讓家屬更信任，還是覺得產品一直不敢給答案
5. 政府與社福單位是否願意接受家庭自行產生的照護事實檔
6. 不接 L0 硬體會不會讓產品被市場誤解成「只是照護任務 app」
7. 「比較不壞」這個冷定位在台灣市場能不能存活，還是市場一定要求溫暖敘事

**這些不能靠思辨解決，必須在 pilot 階段 instrument**。

## 15. 如果你想質疑這份文件

歡迎質疑。但請按以下順序：

1. **先讀完 11 輪原文**：`docs/ux-deep-critique-2026-05-25.md`（1102 行）
2. **指出你想挑戰的是哪一輪、哪一段的論證**——不要泛論
3. **提出你看到、文件裡沒被討論過的新事實或新場景**
4. **如果你的質疑會把產品推回 12 條反功能清單裡，請預先說明為什麼那條反功能應該被例外處理**

「我覺得太冷」「投資人不會買單」「設計師會走」這類質疑**已經被前 11 輪預期到並回應過**。請先確認你的論點不是已經被駁回過的版本。

健康的 dissent 會讓 thesis 更銳利。重複已被駁回的版本只會讓團隊耗能。
